# 阶段 1 — 岗位信息录入

> 当前阶段：1 | 上一阶段：0 | 下一阶段：2 | 可跳过：是

## 本阶段目标

收集目标岗位信息（公司名 + 岗位名 + JD），从JD中提取关键要求。
完成后进入阶段2。如用户没有目标岗位，可跳过。

## 执行步骤

### 1.0 进入提示

```
🎯 岗位信息录入

有目标岗位吗？提供以下任一信息，我将分析岗位需求：
- 粘贴完整 JD（含公司名和岗位名）
- 提供公司名 + 岗位名（我可搜索公开JD）
- 提供招聘链接

[跳过，先不做岗位定向]  [粘贴JD]  [填写公司+岗位]
```

### 1.1 处理JD输入

用户粘贴JD / 提供链接后，使用 WebFetch 获取内容（如有链接），
然后从中提取关键信息：

```
📋 JD 分析：<公司> — <岗位>

🔴 必须项（硬性要求）：
  · <要求1>
  · <要求2>
  ...

🟡 加分项（优先考虑）：
  · <加分1>
  · <加分2>
  ...

🔑 行业 / 技术关键词：
  · <关键词1> · <关键词2> · <关键词3> · ...

🏷 软性线索（团队/文化/业务方向）：
  · <线索1>
  · <线索2>
```

如果用户仅提供公司+岗位名（无JD），尝试通过 WebSearch 搜索该岗位的公开JD信息，
向用户确认搜索到的JD是否与目标一致。

### 1.2 确认JD分析

```
以上分析是否准确？如需补充或修正请说明。

确认后，这些信息将指导后续的数据清洗和简历优化。
[确认]  [修改分析]  [补充JD]
```

### 1.3 保存岗位信息

确认后保存到 `data/input/`：

```bash
# 将JD原始文本 + 分析结果保存
```

文件命名：`data/input/岗位信息-<公司名>-<岗位名>-<日期时间>.md`

保存内容包含：
- 原始JD文本（如有）
- 提取的必须项/加分项/关键词/软性线索
- 来源（用户粘贴 / 链接 / 搜索结果）

### 1.4 完成

发送给阶段2的关键上下文：

```
📌 岗位上下文已锁定：
   公司：<公司名>
   岗位：<岗位名>
   必须项：<N> 项
   关键词：<K> 个

这些信息将在阶段2的数据清洗中自动使用。
```

输出：「✅ 阶段1完成。进入阶段2 — 个人信息收集。」

如果用户选择跳过：
输出：「⏭️ 已跳过岗位信息录入。后续将按通用模式处理。进入阶段2 — 个人信息收集。」

---

## 阶段控制

```
完成 → 进入阶段2（stages/stage-2-input.md）
跳过 → 进入阶段2，后续阶段按通用模式处理
回退 → 从阶段0重新执行
```
