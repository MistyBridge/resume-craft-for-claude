# 阶段 2 — 个人信息收集（岗位导向）

## 本阶段目标

从用户提供的任意信息中提取结构化档案数据。
**如果阶段1已录入岗位信息，数据清洗以JD要求为导向。**
完成并保存后进入阶段3。

## 前置条件

- 阶段1已完成或跳过
- 如果阶段1有岗位信息，本阶段将自动引用岗位上下文

## 执行步骤

### 2.0 告知自动录入方式（进入阶段时首先执行）

```
📥 信息收集

您可以通过以下两种方式提供信息：

[1] 直接输入
    口述经历、粘贴旧简历、提供GitHub仓库链接、指定文件路径等

[2] 自动录入（推荐）
    将您的简历/项目文档/证书等文件放入 data/input/ 文件夹
    然后点击下方「读入」，我将自动扫描并加载

请选择录入方式，或直接开始提供信息。
```

如果用户选择「读入」或直接说读入，进入 2.1。
如果用户直接输入内容，跳到 2.3 处理。

### 2.1 用户触发读入 — 先展示目录，不加载

用户点击「读入」后，**只列出目录，不加载文件内容**：

```bash
ls -lh data/input/
```

将文件列表清晰展示给用户：

```
📂 data/input/ 中发现以下文件：

  1. 我的简历.pdf          (245 KB)
  2. 项目经历.docx         (128 KB)
  3. AWS证书.png           (89 KB)
  4. 工作证明.zip          (1.2 MB)

📌 共 4 个文件。是否全部加载？

[全部加载]  [选择加载（输入编号）]  [取消，我手动输入]
```

**关键**：在用户明确确认之前，**不读取任何文件内容**。

### 2.2 用户确认后 — 逐文件加载处理

用户确认后，按文件类型路由处理。

#### ⛔ 加密文件检测（最高优先级）

读取每个文件前，先检测是否加密：

- **PDF**：打开时提示输入密码 / 内容为乱码密文 → 判定为加密
- **DOCX/XLSX/PPTX**：Office 文档打开时要求密码 → 判定为加密
- **ZIP/RAR/7Z**：解压时要求密码 → 判定为加密
- **其他格式**：打开时报密码/权限错误 → 判定为加密

**检测到加密文件时**：

```
⊘ <文件名> — 🔒 加密文件，已跳过（不尝试破解）
```

处理规则：
- **立即跳过**，不尝试任何解密/破解操作
- 不反复尝试读取
- 收集阶段结束时统一告知用户哪些文件被跳过及原因

#### 文件类型路由

- **文档类**（pdf/doc/docx/pages/odt/txt/md/rtf）→ 提取文本
- **图片类**（png/jpg/jpeg/heic/gif/svg/tiff/bmp/webp）→ 多模态视觉理解
- **表格类**（xls/xlsx/csv/numbers/ods）→ 结构化解析
- **演示类**（ppt/pptx/key/odp）→ 提取文本和结构
- **压缩包**（zip/rar/7z/tar/gz）→ 解压后递归处理内部文件（加密的压缩包跳过）
- **邮件类**（eml/msg）→ 解析邮件头和正文
- **技术开发类**（ipynb/py/java/js/html/css/json/xml）→ 技术栈和代码结构分析
- **音视频类**（mp3/m4a/wav/mp4/mov）→ 提示用户提供文字稿

加载过程中向用户报告进度：

```
⏳ 正在加载...
  ✓ 我的简历.pdf — 提取到 3 段工作经历
  ✓ 项目经历.docx — 提取到 2 个项目描述
  ⊘ 获奖证书.pdf — 🔒 加密文件，已跳过
  ✓ AWS证书.png — 识别到 AWS Solutions Architect
  ⏳ 工作证明.zip — 解压中...
```

#### 加载完成后统一告知

所有文件处理完毕后，汇总跳过情况：

```
📂 文件加载完成

✅ 成功：3 个文件
🔒 跳过：1 个文件
  · 获奖证书.pdf — 文件已加密，请解除加密后重新放入 data/input/

如需重新加载跳过的文件，解密后将文件放入 data/input/ 并说「重新读入」。
```

### 2.3 接收直接输入内容

如果用户选择直接输入（口述、粘贴、链接等），处理方式：
- **文本内容**：直接实体抽取和结构化
- **GitHub 仓库链接**：使用 `gh` CLI 或 WebFetch 获取 README / 技术栈 / 项目定位

**处理后立即备份到 data/input/**（便于跨会话复用）：

备份命名规则：`data/input/<类型>-<日期时间>.md`

| 内容类型 | 文件名示例 |
|---------|-----------|
| 口述经历 | `口述经历-20260513-1430.md` |
| 粘贴的旧简历 | `旧简历文本-20260513-1430.md` |
| GitHub 链接 | `GitHub仓库-20260513-1430.md` |
| 指定文件路径 | `手动指定-20260513-1430.md` |
| 混合输入 | `直接输入-20260513-1430.md` |

备份后告知用户：

```
💾 原始输入已备份至 data/input/口述经历-20260513-1430.md
   下次可直接从 data/input/ 加载，无需重复输入。
```

**注意**：如果用户在 2.2 已通过 data/input/ 文件加载了内容，
此处的直接输入视为**补充内容**，同样备份到 data/input/ 中。

### 2.4 智能提取（岗位导向）

从用户信息中提取结构化数据：

```
提取目标：
├── 基本信息：姓名、目标职位、邮箱、电话、所在地、链接
├── 个人总结：职业定位、核心优势
├── 工作经历：公司/职位/时间/关键成果/技术栈
├── 教育背景：学校/学历/专业/时间/荣誉
├── 技能矩阵：硬技能(含熟练度)/软技能/语言/工具
└── 项目作品：项目名/角色/技术栈/成果/链接
```

#### 🎯 岗位导向提取（如果阶段1有岗位信息）

如果阶段1已录入岗位信息，提取时**动态调整策略**：

**规则 A — 提高JD相关内容的置信度**
JD中明确要求的技能/经历 → 即使置信度偏低也优先提取，标记待确认但不丢弃。

**规则 B — 交叉标注命中情况**
对每条提取结果标注JD命中状态：

```
📋 提取结果：

🟢 已确认 & 命中JD：
  · React — JD要求"精通React" ✓
  · 微信小程序 — JD要求"有小程序开发经验" ✓

🟡 已确认 & 未命中JD：
  · Vue.js — JD未要求，作为附加技能保留

🟡 需要确认：
  1. 您提到"负责核心业务模块"— 具体指什么？
  2. JD要求"有Node.js后端开发经验"，您是否具备？

⚠️ JD要求但您尚未提及：
  · TypeScript（JD硬性要求）
  · CI/CD 经验（JD加分项）
  · 团队管理经验（JD加分项）

🔴 待澄清：
  · 您提到"带过人" — 是管理岗还是指导新人？
```

**规则 C — 主动追问缺失项**
JD硬性要求中用户未提及的能力 → 主动列出并询问：

```
📌 以下JD要求的能力尚未在您的信息中找到，请确认：
  1. TypeScript — JD硬性要求
  2. CI/CD 经验 — JD加分项

如果您具备这些能力请补充，或告知「不具备」跳过。
```

如果阶段1跳过，按通用模式提取（无JD标注和追问）。

置信度分级（通用）：
- 🟢 >90%：直接提取，展示过目
- 🟡 70-90%：展示推断，请确认
- 🔴 <70%：以问题请用户澄清，不给推断

### 2.6 增量补充

用户可以随时补充更多信息。每次补充后更新提取结果。
如发现矛盾信息（如两个不同的入职时间），优先请用户澄清。

每次补充的原始内容同样备份到 `data/input/`，文件名为 `补充输入-<日期时间>.md`。

### 2.7 篇幅大小选择

信息确认完毕后，询问用户简历篇幅目标：

```
📏 选择简历篇幅

请选择目标篇幅，这将影响后续文本的详略程度：

[1] 简略（800-1500字，约1-2页）
    适合：校招海投、招聘会、初筛环节
    特点：高度精炼，仅保留最核心经历，每条经历1-2句

[2] 正常（1500-3000字，约2-4页）
    适合：社招、专业岗位、大多数场景
    特点：标准STAR展开，核心经历详细描述，次要经历精简

[3] 详细（≥3000字，约4页以上）
    适合：学术CV、资深专家、项目申报
    特点：完整展示全部经历，所有项目充分展开

请输入选项编号（默认 [2]）。
```

用户选择后，篇幅设置将影响：
- **阶段3模板设计**：影响学科模板的 max_pages 和 Section 数量
- **阶段4内容填充**：影响每条经历的字数和展开深度
- **阶段5导出**：影响 PDF 分页策略

### 2.8 保存档案

确认后保存为 JSON：`data/profiles/<档案名>.json`

档案 `meta` 字段：

```json
{
  "meta": {
    "length": "brief|normal|detailed",
    "job_context": {
      "company": "<公司名>",
      "role": "<岗位名>",
      "jd_keywords": ["关键词1", "关键词2", ...],
      "must_have": ["必须项1", "必须项2", ...],
      "nice_to_have": ["加分项1", ...]
    }
  },
  "basics": { ... },
  ...
}
```
